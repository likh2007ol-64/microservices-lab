📘 README.md: «Миграция монолита на микросервисы» (учебный проект)

📌 О проекте

Изначально написан «неподдерживаемый» монолит (всё в одном приложении).
Проведена декомпозиция — выделены два независимых микросервиса: user-service (пользователи) и order-service (заказы).
Реализован фича-флаг, который позволяет в любой момент переключить трафик заказов между монолитом и микросервисами.
Проведено нагрузочное тестирование (10 RPS).

🧱 **Краткое описание:**  

Клиент всегда обращается к монолиту (порт 8080).  
Внутри монолита работает переключатель (фича-флаг).

Если флаг выключен (`false`):  
→ монолит сам обрабатывает заказы и сохраняет их в свою БД.

Если флаг включён (`true`):  
→ монолит перенаправляет запросы на создание и получение заказов в `order-service` (порт 8081),  
→ который, в свою очередь, проверяет пользователей через `user-service` (порт 8082).

Каждый сервис имеет свою базу данных H2.

🛠️ Технологии

Java 17 (можно использовать 21)
Spring Boot 3.2.4
H2 Database (in-memory, для простоты)
Maven (сборка)
REST API (взаимодействие между сервисами)
IntelliJ IDEA (разработка)

🚀 Как запустить проект

1. Убедитесь, что у вас установлены:
   Java 17 или выше (проверьте: java -version)
   Maven (проверьте: mvn -version)

2. Соберите проект:
   Откройте терминал в корневой папке и выполните:
   bash
   mvn clean package
   В папках monolith/target, order-service/target, user-service/target появятся JAR-файлы.

3. Запустите сервисы (важен порядок!):
   Сначала user-service (порт 8082):
   bash
   java -jar user-service/target/user-service-1.0-SNAPSHOT.jar
   Затем order-service (порт 8081):
   bash
   java -jar order-service/target/order-service-1.0-SNAPSHOT.jar
   Затем монолит (порт 8080):
   bash
   java -jar monolith/target/monolith-1.0-SNAPSHOT.jar

💡 Можно запускать и из IDEA, но для демонстрации удобнее использовать JAR-файлы.

🔧 Как переключать фича-флаг

Фича-флаг управляет тем, где обрабатываются заказы.

Настройка находится в файле:
monolith/src/main/resources/application.properties

Строка: feature.use-microservices=false (по умолчанию выключено)

Чтобы переключить:
Остановите монолит.
Измените значение на true (или false).
Сохраните файл.
Запустите монолит заново.

⚠️ Изменения вступают в силу только после перезапуска монолита.

🧪 Как тестировать (самое важное!)

Для тестирования создан файл test-requests.http в корне проекта. 
В IntelliJ IDEA можно открыть этот файл и выполнять запросы, нажимая 
зелёные стрелки ▶️ напротив каждого запроса.

📋 Универсальный тестовый сценарий (работает при любом значении флага)

▶️ Создать пользователя (обязательно, чтобы order-service мог его проверить):

http
POST http://localhost:8082/users
Content-Type: application/json
{
"name": "Иван Петров",
"email": "ivan@example.com",
"address": "ул. Ленина, д. 1"
}
✅ В ответе вы получите id=1.

️▶ Создать заказ через монолит:

http
POST http://localhost:8080/orders
Content-Type: application/json
{
"pizza": "Маргарита",
"userId": 1,
"status": "PENDING"
}
✅ Заказ создастся (либо локально, либо через order-service — зависит от флага).

▶ Получить заказ через монолит:

http
GET http://localhost:8080/orders/1
✅ Вернёт созданный заказ.

📊 Что смотреть в логах (консолях)?

false (выключен):	
Консоль монолита 📊 Monolith RPS: 1
Консоль order-service	📊 OrderService RPS: 0 (или ничего)

true (включён):	
Консоль монолита 📊 Monolith RPS: 1 (входящий запрос)
Консоль order-service	📊 OrderService RPS: 1 (подтверждение перенаправления)
Консоль user-service (опционально)	📊 UserService RPS: 1 (если order-service вызвал его)

🧪 Проверка переключения (пошагово)

Запустите все три сервиса. 
Убедитесь, что флаг в monolith/application.properties стоит false.
Перезапустите монолит.
Выполните все три запроса из универсального сценария (шаги 1–3).
Посмотрите логи — заказы обработаны внутри монолита (только 📊 Monolith RPS).
Переключите флаг на true, перезапустите монолит.
Снова выполните те же запросы (пользователя можно не создавать заново, если он уже есть в user-service).
Теперь в логах order-service появятся 📊 OrderService RPS: 1 — значит, монолит перенаправил запрос.

⚡ Нагрузочное тестирование (10 RPS)

В проекте есть класс LoadTestRunner (в модуле monolith), который отправляет 10 запросов в секунду на создание заказа 
в течение 30 секунд.

Чтобы запустить:

Убедитесь, что все сервисы запущены и пользователь создан в user-service.
Запустите класс LoadTestRunner как обычное Java-приложение (правой кнопкой → Run).
Наблюдайте за логами — каждую секунду в консолях монолита и order-service будет появляться 📊 ... RPS: ~10.
После завершения теста вы увидите итоговое количество отправленных запросов.
Результат: нагрузка 10 RPS стабильно выдерживается — цель достигнута.

❓ Частые вопросы

Вопрос: Почему при тесте с флагом true возникает ошибка 500?
Ответ: Скорее всего, пользователь с указанным userId отсутствует в user-service. 
Обязательно выполните запрос на создание пользователя в user-service (порт 8082).

Вопрос: Нужно ли перезапускать user-service и order-service при смене флага?
Ответ: Нет, перезапускается только монолит. Микросервисы могут работать постоянно.

Вопрос: Где хранятся данные после перезапуска?
Ответ: Используется H2 in-memory, поэтому при остановке сервиса все данные теряются. 
Для тестов это удобно — всегда чистая база.
